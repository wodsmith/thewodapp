Root cause confirmed — it's a data-model bug, not just one submission path

  The issue is structural and affects any view that reads scores by userId.

  The broken invariant

  scoresTable has a unique index on just (competitionEventId, userId) — apps/wodsmith-start/src/db/schemas/scores.ts:122-125. A single user can only have one score row per competition event,
  regardless of division. When an online competition reuses the same trackWorkout across both an individual and a partner division, a user registered in both divisions has one shared row. Every
  write is onDuplicateKeyUpdate, so the last submission wins and overwrites scalingLevelId.

  Where the bug manifests

  All read paths key scores by (competitionEventId, userId) with no division filter, so whichever row exists is attributed to every registration the user has:

  - Preview leaderboard — apps/wodsmith-start/src/server/competition-leaderboard.ts:740-744 fetches by userIds, then at lines 882-893 maps each score back via filteredRations.find(r =>
  r.user.id === score.userId) and buckets into registration.registration.divisionId (NOT score.scalingLevelId). When the leaderboard is filtered to the individual division, it finds the individual
  registration and the partner-division score lands there.
  - Organizer athlete detail page (…/athletes/creg_…) — apps/wodsmith-start/src/server-fns/organizer-athlete-fns.ts:366-374 queries inArray(userId, …) AND inArray(competitionEventId, …) with no
  division filter.
  - getAthleteEventScoreFn — athlete-score-fns.ts:562-568 matches (competitionEventId, userId) only.

  Contributing bugs in write paths

  - athlete-score-fns.ts:359-375 (submitAthleteScoreFn) does registration lookup with .limit(1) and no divisionId filter — whichever registration returns first wins. The input schema also doesn't
  accept a divisionId (line 85-92). This file looks orphaned (not referenced anywhere), but still live code.
  - video-submission-fns.ts:1113-1147 and routes/api/compete/scores/submit.ts:2ectly scope the registration lookup by divisionId, but the upsert still collides on (competitionEventId,
  userId), so the row still gets clobbered.

  Recommended fix (not implemented yet — want your sign-off)

  1. Change the unique index to (competitionEventId, userId, scalingLevelId) so one user can hold a distinct row per division. This is the root fix.
  2. Read paths: filter scores by scalingLevelId = registration.divisionId in competition-leaderboard.ts (fetchScores + the grouping at line 890 should use score.scalingLevelId as ground truth),
  organizer-athlete-fns.ts, and getAthleteEventScoreFn.
  3. Write paths: require divisionId in submitAthleteScoreFn and /api/compete/scores/submit input, scope the registration lookup by it, and use onDuplicateKeyUpdate that matches the new 3-column
  unique key.
  4. Data backfill: existing score rows already have scalingLevelId, but we need to confirm no data will collide under the new index (it can't — only one row existed before).

  Tradeoff: This needmigration (pnpm db:generate) and touches several read/write sites. The alternative — adding an explicit registrationId FK to scoresTable — is cleaner but a bigger change.
